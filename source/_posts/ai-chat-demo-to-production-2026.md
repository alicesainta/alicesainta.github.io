> 很多 AI 对话项目都停在 Demo 阶段：页面能聊天，但一上量就不稳定、不可追踪、不可维护。本文给一条从 Demo 到可上线的最小改造路径。

## 一、为什么“前端直连 LLM”只适合 Demo

前端直连大模型 API 在早期验证阶段很快，但上线会暴露几个硬问题：

- 密钥风险：密钥暴露在浏览器端，无法真正保护。
- 成本风险：无法做服务端限流和配额管理，容易失控。
- 合规风险：无法统一做日志审计和敏感数据治理。
- 稳定性风险：重试、熔断、降级都很难在前端统一处理。

## 二、上线架构的最小形态

建议先落这个最小链路：

`Browser -> Backend API -> LLM Provider -> Database / Vector DB`

职责分层：

- Browser：交互、输入、流式渲染、用户态体验。
- Backend API：鉴权、限流、审计、Provider 适配、业务编排。
- Database：用户、会话、消息、配置持久化。
- Vector DB：知识库向量存储与检索召回。

## 三、后端代理层到底在解决什么

后端代理不是“多一跳”，而是把工程能力集中起来。

核心能力清单：

1. 统一鉴权：校验用户身份、租户权限、接口权限。
2. 统一限流：按用户/模型/租户做速率控制。
3. 统一调用：封装不同 Provider 的请求协议与错误码。
4. 统一审计：记录请求上下文、token 用量、错误轨迹。
5. 统一降级：上游超时或失败时切换模型、返回兜底响应。

## 四、会话持久化：先把最小数据模型打通

先不要追求复杂 schema，最小可用就够：

- `users`：用户信息、角色、状态
- `conversations`：会话标题、归属用户、创建时间
- `messages`：角色、内容、token、耗时、错误状态
- `attachments`（可选）：文件元数据和存储引用

最重要的不是“字段多”，而是“链路完整”：

- 创建会话 -> 发送消息 -> 流式返回 -> 落库终态

## 五、RAG 最小可用链路（MVP）

很多项目只做了“上传文档 + UI”，这不算可用的 RAG。  
最小可用链路至少包含：

1. 文档清洗：去噪、切段、标准化编码。
2. 分块（chunking）：固定窗口 + overlap 起步。
3. 向量化（embedding）：写入向量库并保存文档元数据。
4. 检索召回：按 query 召回 topK 候选片段。
5. 可选重排（rerank）：提升相关性质量。
6. 上下文拼接：控制 token 预算并构造提示词。
7. 生成回答：返回结果并标注来源片段（可选）。

你不需要一次做满高级特性，但要确保每一步都“真实可跑”。

## 六、流式输出性能：避免 token 一来就 setState

常见错误：每个 token 到来就直接更新 UI，导致高频重渲染。

更稳妥的做法：

- 前端维护缓冲区，按固定节奏批量 flush（如 16~50ms）。
- 把状态机明确为：`submitted -> streaming -> completed/failed/aborted`。
- 只有终态时才做落库确认、按钮恢复、资源清理。

这样能明显降低卡顿和“卡在 loading”类问题。

## 七、安全与稳定性：上线前必须补的防线

建议最小补齐：

- 提示注入防护：限制系统指令边界，不信任外部文本。
- 超时与重试：请求超时、指数退避、幂等设计。
- 错误分级：可恢复错误与不可恢复错误分开处理。
- 监控指标：成功率、P95 延迟、TTFT、token 消耗、错误码分布。
- 审计日志：谁在什么时候用哪个模型做了什么请求。

## 八、CI/CD 与发布门禁

把质量门槛前移到 PR 阶段：

1. `format`
2. `lint`
3. `test`（核心链路至少有基础覆盖）
4. `build`

失败即阻断合并，避免“线上才发现问题”。

## 九、上线前检查清单

- [ ] 前端不再直连 LLM，全部走后端代理
- [ ] 会话与消息已持久化，不再只依赖浏览器本地存储
- [ ] RAG 链路至少可完成“上传 -> 向量化 -> 召回 -> 回答”
- [ ] 流式输出采用缓冲与终态状态机
- [ ] 有统一日志、错误码和监控面板
- [ ] CI 启用质量门禁（lint/test/build）

## 十、常见反模式

- “功能看起来有”但没有真实后端链路。
- “知识库做了”但没有向量检索落地。
- “能聊天”但没有审计、限流和失败恢复。
- “本地能跑”但团队环境和 CI 结果不一致。

## 总结

把 AI 对话 Demo 做成可上线项目，不是重写一遍，而是补齐工程闭环。  
先把最小链路打通，再逐步增强模型能力和产品体验，这样迭代速度和稳定性都会更高。
